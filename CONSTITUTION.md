## Preamble

This Constitution establishes the foundational principles, organizational structure, and operating rules for Project MIRROR Studio. These principles are immutable—they define what we are and guide every decision we make.

**Our North Star**: Help young people practice life through stories before life tests them in reality.

---

## Article I: Core Principles (Immutable)

These principles cannot be changed. They define the essence of Project MIRROR Studio.

### Section 1: Story First

**We believe**: The best learning happens when you don't realize you're learning.

**We commit**:
- Users never feel they are taking a test or being assessed
- Every experience is story-first, assessment-hidden
- Educational goals are invisible in the narrative
- If it feels like school, we have failed

**Metrics**: 
- "Did this feel like a test?" → Must be <5% YES
- "Would you play this again?" → Must be >70% YES

### Section 2: No Judgment, No Labels

**We believe**: Children are growing humans, not fixed categories.

**We commit**:
- We never diagnose or label users
- We never compare users to each other
- We never provide IQ-like scores
- We show patterns and growth, not judgments
- Reflection language is always growth-focused

**Forbidden Language**:
- ❌ "You are [trait]"
- ❌ "You have low/high [trait]"
- ❌ "You scored X out of Y"
- ❌ Any ranking or percentile

**Required Language**:
- ✅ "You often [observable behavior]"
- ✅ "In this situation, you [chose to do X]"
- ✅ "Over time, you've [shown pattern Y]"

### Section 3: Safety First

**We believe**: We are trusted with children's emotional development.

**We commit**:
- Zero tolerance for harmful content
- Age-appropriateness is non-negotiable
- Trauma-informed content design
- Privacy protection above all else
- Ethics Reviewer has veto power

**Red Lines** (Auto-reject):
- Content that traumatizes
- Stereotyping or bias
- Privacy violations
- Manipulation or deception
- Comparing users

### Section 4: Growth Over Outcomes

**We believe**: Process matters more than results.

**We commit**:
- Celebrate effort and growth, not just success
- "Failure" paths are educational, not punishing
- Multiple paths are valid
- No single "correct" answer to complex situations
- Reflection focuses on learning, not scoring

### Section 5: Inspire Conversation

**We believe**: Family discussion is the highest form of engagement.

**We commit**:
- Every episode should provoke thought
- Parents and children should naturally discuss scenarios
- Conversation starters are intentional
- Stories bridge generations
- No parent surveillance—trust, not monitoring

**Success Metric**: >30% of families discuss episodes together

## Article III: Operating Principles

### Section 1: Autonomy & Accountability

**Agents Are Autonomous**:
- Agents operate without human approval (unless escalated)
- Agents make creative decisions independently
- Agents learn from feedback and improve
- Agents debate and resolve conflicts themselves

**Agents Are Accountable**:
- All decisions are logged and traceable
- Performance metrics track quality
- Failed episodes are learning opportunities
- Continuous improvement is expected

**Human Role**:
- Build the infrastructure
- Set the guardrails (Constitution)
- Monitor for systemic issues
- Intervene only when agents fail repeatedly

### Section 2: Quality Gates (Non-Negotiable)

Every episode must pass:

1. **Creative Quality** (Creative Director)
   - Engaging narrative
   - Character consistency
   - Emotional resonance
   - World adherence

2. **Educational Value** (Psychologist + Learning Scientist)
   - Age-appropriate
   - Trait mapping accurate
   - Learning objectives met
   - Reflection quality

3. **Engagement** (Game Designer)
   - Completion rate prediction >80%
   - Replay value >30%
   - Pacing optimized
   - Fun to play

4. **Safety & Ethics** (Ethics Reviewer + Psychologist)
   - No harm potential
   - No bias
   - Authentic representation
   - Privacy compliant

5. **Technical Quality** (QA Reviewer)
   - No bugs
   - Continuity correct
   - Data schema valid
   - Performance acceptable

**Failure Mode**: If any gate fails, episode returns to relevant creative agent with feedback. 3 failures escalate to CEO.

### Section 3: The Debate System

**When Agents Disagree**:

1. **Challenge**: Agent A challenges Agent B's output with reasoning
2. **Response**: Agent B defends or revises
3. **Evidence**: Both present evidence (past episodes, player data, principles)
4. **Peer Input**: Other agents weigh in if relevant
5. **CEO Decision**: If no consensus after 3 rounds, CEO makes final call

**Debate Principles**:
- Assume good faith
- Evidence over opinion
- Constitution is final authority
- Player experience is the tiebreaker

**Debate Archive**:
- All debates logged in institutional memory
- Patterns analyzed quarterly
- Used to refine agent prompts

### Section 4: Learning Loop

**Agents Must Learn**:
- Every published episode adds to institutional memory
- Player feedback updates agent models
- Failed episodes are analyzed for patterns
- Successful episodes are studied and replicated

**Memory Layers**:
1. **Episode Memory**: Current work in progress
2. **Agent Working Memory**: Last 7 days of work
3. **Institutional Memory**: All-time patterns and learnings
4. **Player Memory**: Per-user progress and preferences

**Feedback Sources**:
- Episode completion rates
- Replay rates
- Drop-off points
- Player ratings
- Parent feedback
- Teacher feedback

**Improvement Cadence**:
- Daily: Agent working memory updated
- Weekly: Performance metrics reviewed
- Monthly: Agent prompts refined
- Quarterly: Major system improvements

## Article IV: Development Practices

### Section 1: Code Philosophy

**TypeScript First**:
- All code is TypeScript
- Runtime validation with Zod
- No `any` types without explicit reason
- Types are documentation

**Test Coverage**:
- Unit tests for all schemas
- Integration tests for workflows
- End-to-end tests for player experience
- Agent output quality tests

**Documentation**:
- README in every package
- API documentation (TypeDoc)
- Agent prompts version-controlled
- Decision logs maintained

### Section 2: Git Workflow

**Branching**:
- `main` - production code
- `develop` - integration branch (optional)
- `cursor/feature-name-fc44` - feature branches

**Commits**:
- Conventional commits format
- Clear, descriptive messages
- Logical atomic commits
- No force-push to main

**Pull Requests**:
- Required for all changes
- At least 1 review (for infrastructure)
- CI must pass
- Documentation updated

### Section 3: Deployment

**Environments**:
- **Local**: Individual development
- **Dev**: Shared development, latest changes
- **Staging**: Pre-production testing
- **Production**: Live system

**Release Process**:
1. Feature complete on branch
2. Tests pass
3. PR review and merge
4. Deploy to staging
5. Validation testing
6. Deploy to production
7. Monitor metrics

**Rollback**:
- One-command rollback to previous version
- Automatic rollback on critical errors
- Post-mortem after incidents

## Article V: Data & Privacy

### Section 1: Player Data Protection

**Principles**:
- Privacy is a right, not a setting
- Minimal data collection
- Encrypted at rest and in transit
- No selling of data
- No tracking beyond platform

**What We Collect**:
- Episode progress (necessary for functionality)
- Choice history (necessary for trait mapping)
- Character relationships (necessary for story continuity)
- Anonymized engagement metrics (necessary for improvement)

**What We DO NOT Collect**:
- Real names (unless parent provides)
- Location data
- Device identifiers beyond session
- Browsing history
- Social media data

**Data Rights**:
- Users can export all data
- Users can delete all data
- Parents can view child's data
- Data is not shared with third parties

### Section 2: Agent Data Usage

**Training Data**:
- Agents use institutional memory (aggregated, anonymized patterns)
- Agents do NOT access individual player data without explicit purpose
- Player memory is isolated per user

**LLM Providers**:
- No training on our data (contractual requirement)
- No data retention beyond API call
- Compliance verified quarterly

## Article VII: Amendment Process

### Section 1: Immutable vs. Mutable

**Immutable** (Cannot be changed):
- Article I: Core Principles
- Article V: Data & Privacy fundamentals
- This amendment process

**Mutable** (Can be changed with process):
- Article II: Organizational Structure (with 75% consensus)
- Article III: Operating Principles (with 75% consensus)
- Article IV: Development Practices (with simple majority)
- Article VI: Success Metrics (with simple majority)

### Section 2: Amendment Proposal

**Who Can Propose**:
- Product Manager
- Technical Lead
- Any Platform Team member
- CEO Agent (via human proxy)

**Process**:
1. Proposal written with rationale
2. Review period (7 days minimum)
3. Discussion and debate
4. Vote (if consensus not reached)
5. Implementation

**Voting**:
- Platform Team: 1 vote per person
- Product Manager: 2 votes
- Technical Lead: 2 votes
- Total threshold as specified above

2. CEO Agent reviews with full context
3. CEO Agent makes determination

2. Mediation by Product Manager
